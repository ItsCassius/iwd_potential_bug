Subject: [iwd 3.12] connect-timeout on boot blacklists BSSID; rank threshold
         prevents autoconnect recovery without daemon restart

================================================================================
POTENTIAL BUG REPORT — iwd 3.12-1
================================================================================

NOTE: I used Claude code to attempt to figure this out (I am not a kernel or 
driver developer). I did audit it, spending hours testing and retesting things.
So what's here isn't just hot mess vibe coding response, it's somewhat polished
hot mess. You can reach me on matrix @itscassius:matrix.org if you have any
questions.

TLDR
----

After updating from 3.11-2 -> 3.12-1 I was not connecting to my usual 5GHz AP.
Seems to be related to signal strength poisoning the reconnectability. I would
like to help fix this if it is an issue, but I'm not sure if it's actually an
issue.

SUMMARY
-------
On boot, iwd attempts to connect to a known 5GHz network. The AP sends a deauth
mid-handshake (transient AP-side boot race), resulting in connect-timeout
reason: 3 (NETDEV_RESULT_FAILED). The BSSID is blacklisted by src/blacklist.c.

When the blacklist expires, the BSS rank has settled below iwd's autoconnect
threshold (rank 245). iwd permanently filters it as "No suitable BSSes found"
and never retries. The stuck state also blocks manual iwctl connection attempts.
Daemon restart is required to recover.

This network and signal level connected reliably on every boot with iwd 3.11-2.


ENVIRONMENT
-----------
Distribution:  Arch Linux
iwd version:   3.12-1   (regression from 3.11-2)
Kernel:        6.18.21-1-lts
NetworkManager: 1.56.0 (wifi backend: iwd)
Wireless NIC:  phy0 — supports 2.4GHz (HT40) and 5GHz (HT40/VHT80)
Adapter: Intel 7265


STEPS TO REPRODUCE
------------------
Requirements:
  - iwd 3.12 managing a known 5GHz WPA2-PSK network
  - Signal at approximately -72 to -77 dBm at boot (2-bar signal)
  - AP may still be initialising when iwd first attempts connection (boot race)

1. Ensure iwd has a valid .psk file for a known 5GHz network.

2. Enable iwd debug logging:
     sudo tee /etc/systemd/system/iwd.service.d/debug.conf <<'EOF'
     [Service]
     ExecStart=
     ExecStart=/usr/lib/iwd/iwd -d
     EOF

     sudo systemctl daemon-reload

3. Reboot. Do not manually intervene. Wait ~3 minutes.

4. Check autoconnect result:
     journalctl -b -u iwd --no-pager | grep -E \
       'connect-timeout|connect-failed|No suitable|blacklist|autoconnect'

5. Attempt manual connection:
     iwctl station wlan0 connect "MyRouter-5G"

EXPECTED: iwd recovers from the transient boot failure and connects on retry,
          as it did with iwd 3.11-2 at the same signal level.

ACTUAL:   iwd blacklists the BSSID after reason: 3. When blacklist expires,
          BSS rank (245) is below autoconnect threshold — permanently filtered.
          Manual iwctl also returns "Operation failed" (BSSID re-blacklisted
          after second attempt). Requires 'systemctl restart iwd' to recover and
          requires that the attempted reconnect has a stronger signal strength.

Recovery behaviour after 'sudo systemctl restart iwd':

  The restart clears two in-memory structures:
    1. The BSSID blacklist  — unblocks manual iwctl connection attempts
    2. The PMKSA cache      — prevents the bad PMKID being sent to the AP again

  Observed:
    - Manual 'iwctl station wlan0 connect' fails after restart regardless
      of signal strength.
    - Autoconnect only resumes if BSS rank is above the threshold on the fresh
      scan (i.e. signal needs to be stronger than ~-74dBm for rank > 245).

WHAT CHANGED FROM 3.11 TO 3.12
--------------------------------
With iwd 3.11-2, this network and signal level connected reliably on every boot.

iwd 3.12 changelog includes two relevant fixes:
  - PMKSA expiration handling fixes
  - Buffer/PMKID initialization fix

SUSPECTED ROOT CAUSE: The PMKID change is the likely trigger for the initial
failure. The debug log shows 'Del Station(20)' appearing between the Authenticate
and Associate MLME notifications — this means the AP sent a deauth frame
mid-handshake, before the 4-way EAPOL exchange even began:

  MLME notification Authenticate(37)   ← OK
  MLME notification Del Station(20)    ← AP deauths the client
  MLME notification Associate(38)
  MLME notification Connect(46) aborting
  connect-timeout, reason: 3 (NETDEV_RESULT_FAILED)

If iwd 3.12's PMKID fix changed what is sent in the Association Request (e.g.
a previously zero-initialized PMKID is now populated from a stale PMKSA cache
entry), the AP may reject it and deauth, producing exactly this sequence.

Note: src/blacklist.c is present and unchanged in both 3.11 and 3.12. The
blacklist and rank threshold behaviour are not themselves regressions — they
are pre-existing mechanisms exposed by the new connection failure pattern.
The regression is the initial AP deauth (reason: 3) that was not present in
3.11, which then triggers blacklisting and prevents autoconnect recovery.
